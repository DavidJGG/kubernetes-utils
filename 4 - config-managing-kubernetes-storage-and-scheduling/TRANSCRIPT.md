### Why Persistent Storage Is Critical

Hi, and welcome to Configuring and Managing Kubernetes Storage and Scheduling here at Pluralsight. My name is Patrick Rusch, a Senior Consultant and IT instructor from Germany, and I am thrilled to take you on this journey. In this course, we're going to explore two of the most important parts of running real‑world Kubernetes applications, persistent storage and intelligence scheduling. You'll learn how to configure storage for your workloads, manage how Kubernetes schedules your pods, and make sure your apps run efficiently and reliably. We'll start with the basics, dive into real‑world examples, and see how these concepts apply to everyday development challenges. Throughout the course, you'll see short demos that show exactly how things work inside Kubernetes cluster. Meet Steve, a DevOps engineer at Globomantics. His team has recently containerized several internal applications, databases, analytics tools, and small APIs, all running smoothly in Kubernetes. But after each new deployment, something strange happens. Their applications lose data. Logs disappear, uploaded files vanish, and Steve keeps getting calls from developers wondering where their data went. What's happening here? This is one of the first real‑world problems Kubernetes admins encounter. Understanding that by default data inside containers is ephemeral. When a pod goes away, so does its data. When we talk about containers, one of their biggest strengths is how lightweight and disposable they are. You can deploy them, replace them, and move them around the cluster without worrying about the underlying host. But that design also comes with a trade‑off. Containers are stateless by default. Every time a pod is restarted, replaced, or rescheduled to a different node, anything stored in the container's file system disappears. For many workloads, that's fine. A simple web frontend or API service can start fresh every time because it doesn't rely on local data. But what about applications that need to keep state? Like database, a file upload service, or even a background job that writes logs. These need data that survives restarts, updates, and even node failures. And that's where persistent storage comes in. Persistent volumes allow Kubernetes to maintain data independently of the pod that use it. So even if the pod is deleted or recreated, the data stays safe and available. In other words, persistent storage brings durability and reliability to workloads that would otherwise lose everything when a container stops. It's one of the key building blocks that turns Kubernetes from a platform for stateless microservices into one that can also run stateful production‑grade applications. Now that we understand why persistent storage matters, let's look at how Kubernetes actually handles it, through persistent volumes, or PVs, and PersistentVolume claims, or PVCs. A PersistentVolume represents a piece of storage that exists in the cluster, provisioned either by an administrator or dynamically through storage classes. You can think of it as a managed disk, NFS share, or any external storage resource that Kubernetes knows how to talk to. It's part of the cluster's infrastructure, available to be claimed and used by workloads when they need it. On the other side, we have PersistentVolume claims. This is what developers or workloads use to request storage. Instead of worrying about where that storage comes from or how it's created, a PVC simply says, I need 5 GB of ReadWriteOnce storage. And Kubernetes then acts as the matchmaker. It finds or provisions a PersistentVolume that fits the request and binds it to the claim. Once that binding happens, the pod can mount the claim just like a regular volume, read and write data, and rely on it to persist even if the pod itself is replaced. And this separation of concerns is really powerful. Cluster admins can manage the underlying storage infrastructure and developers just request the storage they need. The result is flexibility, reusability, and consistency, and that's exactly what we need in a production‑grade Kubernetes environment. Let's start with the foundation, the PersistentVolume itself. In this example, we are defining a simple PersistentVolume using a host path which stores data directly on the node's file system. That makes it great for local testing or running Kubernetes on Docker Desktop. The key fields here are the capacity and accessMode. We're giving this volume 1 GB of space and marking it as ReadWriteOnce, meaning it can be mounted by a single node at a time. Notice we also define a storageClassName. This helps Kubernetes identify which storage class this volume belongs to. In this case, we're setting it to local‑storage, which could correspond to manually managed disks or directories. In a production environment, you'd likely use a network‑backed or dynamically proficient volume. For example, something based on an AWS, EBS, NFS, or another backend. But for learning purposes, hostPath makes it easy to see how Kubernetes ties everything together. And once we apply this manifest, Kubernetes registers the PersistentVolume and marks it as available, waiting for a claim that matches its size, access mode, and storage class. Now that we have a PersistentVolume ready, the next step is to create a PersistentVolumeClaim, the developer's way of requesting storage. Here, the PVC is asking for exactly 1 GB of storage with the same accessMode and storageClassName that your volume provide. Once we apply this claim, Kubernetes looks for an available PersistentVolume that matches the criteria. If it finds one, and it will since we just created it, it automatically binds the claim to that volume. You can verify this binding by running kubectl get pvc, which will show the claims status at bound. From this point on, any pod that references this claim will get access to the underlying storage. That means data written by one pod stays available to others, even if the pod is restarted, rescheduled, or replaced. And this is what makes persistent volumes and claims so powerful. They decouple storage from compute, allowing applications to remain flexible and reliable without losing their data.

### StorageClasses and Dynamic Provisioning

As Globomantics continues to scale, Steve's team is onboarding new microservices almost every week. Each of these services needs its own persistent storage for databases, logs, or user uploads, and that quickly turns into a management nightmare when every PersistentVolume has to be created manually. It's not just time consuming; it's error‑prone and inconsistent. One developer might use a slow disk, another might forget to set the reclaim policy, and before long, storage management becomes a bottleneck. Kubernetes solves this elegantly with storage classes, which act like blueprints for storage. Instead of Steve or his ops team manually provisioning volumes, Kubernetes automatically creates the right kind of storage whenever a developer requests it through a PersistentVolumeClaim. And this approach keeps the developer experience simple, just requests what you need while giving the platform team centralized control over performance, cost, and lifecycle policy. Let's take a closer look at how this works. Think of a storage class as a recipe for storage. It defines the how, the provisioner that creates the storage, along with parameters like disk type, replication, or performance class. When a developer creates a PersistentVolumeClaim and specifies that storage class, Kubernetes automatically provisions a matching PersistentVolume, binds it to the claim, and mounts it to the pod. Behind the scenes it's Kubernetes negotiating with the underlying storage backend, whether that's cloud disk, NFS shares, or local parts, or without the developer ever needing to touch the infrastructure layer. And the result is dynamic provisioning, storage that created only when needed and tailored exactly to the application's requirements. This model brings the same kind of agility to storage that Kubernetes already provides for a compute. And here's an example of a simple storage class definition. We start by giving it a name. In this case, fast‑storage. The provisioner field tells Kubernetes how to create the underlying storage. In production clusters, you'd likely use a container storage interface, or a CSI driver, something like ebs.csi.aws.com for AWS or csi.vsphere.vmware.com for VMware environments. But for our local Docker Desktop setup, we'll use the built‑in kubernetes.io/no‑provisioning option which lets us work without any external storage backend. Next, the volumeBindingMode controls when volumes are created. The Immediate mode means Kubernetes binds the volume as soon as the claim is made. And finally, the reclaimPolicy determines what happens to the underlying volume when the PVC is deleted. Setting it to delete is perfect for dev and test workloads, since it automatically cleans up unused resources, keeping Steve's cluster tidy without manual intervention. For production workloads, though, Steve might switch to retain to prevent accidental data loss. And now that the storage class is defined, developers can use it simply by referencing its name in their PVCs. Here, the PVC called demo‑pvc requests 1 GB of storage with ReadWriteOnce access, meaning it can be mounted by a single node at a time. The key line is storageClassName: fast‑storage, and this tells Kubernetes which blueprint to use when creating the actual PersistentVolume. Once applied, Kubernetes automatically provisions a volume that matches the claim, binds it to the PC, and makes it available to the pod. The developer doesn't need to know anything about the underlying disks or network storage. Kubernetes abstracts all of that complexity away. This not only streamlines developer workflows, but also ensures consistent, policy‑driven storage behavior across every microservices in Steve's cluster. In short, storage classes help teams scale storage management with the same automation and flexibility that Kuperittis brings to the containers themselves.
