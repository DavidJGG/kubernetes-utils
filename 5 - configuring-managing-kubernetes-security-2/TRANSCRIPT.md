

### Introducing Authentication and RBAC

Securing access to your Kubernetes cluster is the foundation of Defense in Depth. You need to control who can access the Kubernetes API and what they can do with your resources, otherwise every other security measure can be bypassed. We'll start with authorization. Kubernetes uses role‑based access control to manage permissions. You define what access is available, whether you can delete pods or read config maps. Those go into a set of rules, and you also define who the rules apply to. That all lives in your application model with your other YAML specs. There are four resource types you need to know. Roles define a set of allowed permissions. A role contains rules that specify actions as verbs, get, list, create, delete. Those are combined with resources, pods, services, secrets. You might create a role called pod‑reader that allows getting and listing pods, but nothing else. Roles are namespace scoped, they express access levels to objects inside the same namespace where the role is deployed. RoleBindings connect roles to security principals, either service accounts or users. A RoleBinding says this principal gets the permissions from this Role in this namespace. The binding is what actually grants the permission. You can have multiple bindings to the same role, granting the same permissions to different accounts and multiple bindings per account, giving principals several sets of permissions. ClusterRole's work like Roles, but have two purposes, they can define permissions for cluster‑wide resources like nodes or namespaces themselves, they can also serve as templates for permissions that can be used across multiple namespaces. Kubernetes includes default ClusterRoles like view, edit, and admin that provide standard permission sets. There's also cluster‑admin, which grants everything on every resource, useful for break‑glass scenarios when you need to fix something quick, but dangerous for everyday use. ClusterRoleBinding's, grant ClusterRole permissions across the entire cluster. Bind the view ClusterRole to a monitoring service account, it can read resources in every namespace. Bind cluster‑admin to a user, they become root for your entire cluster. You can also use a RoleBinding with a ClusterRole, you could give a user view access to the test namespace, but they may have admin for their own namespace. Every API request gets evaluated against these RBAC rules, it could be an app inside the cluster making REST calls to the Kubernetes API server or from a remote user running kubectl commands. Kubernetes identifies the authenticated caller, the security principal. Next, it finds all relevant RoleBindings and ClusterRoleBindings, and combines the permissions. Then it makes an allow or deny decision, no permissions means no access. Wired Brain Coffee failed their latest security audit because they ignored RBAC. Service accounts were running with admin access because someone set that up for debugging and never removed it. End users with cluster‑admin roles, because that's the easiest thing to set up, no way to expire accounts, so once someone is given access, they always have access, those are classic security anti‑patterns that attackers exploit daily, and it's easy to address them.

### Understanding End-user Authentication

You need to have a good understanding of RBAC, because the default permission set up in Kubernetes isn't necessarily secure. We've seen how the Wired Brain Team accidentally gave every pod cluster‑admin access by misconfiguring the default service account, then fixed it with dedicated accounts and least privilege RBAC. Service accounts secure access from inside the cluster when applications need to talk to the API using a token to authenticate. But most applications don't need API access at all, application pods mostly just serve their function, they don't manage in cluster resources, so they don't need to authenticate. For these workloads, you should disable ServiceAccount token mounting entirely, set automountServiceAccountToken to false on the Pod spec, or better still on the ServiceAccount. No token means no API access, even if someone compromises the container. End user access, humans connecting with kubectl, is different. Kubernetes doesn't manage users internally, there's no User object to create, instead, it trusts external systems to vouch for identity through certificates, tokens, or custom authentication, and then RBAC takes over for authorization. Certificate‑based authentication is what you get by default. The cluster CA signs a certificate with the username in the common name field, simple, works everywhere, but certificates can't be revoked individually, and there's no multi‑factor authentication. Once someone has that certificate, they have access until it expires, which might be years. Certificates, or static tokens, get embedded in the config file, which kubectl uses for cluster connections, so it's basically a plain text password file. If that file gets compromised, then an attacker has all the access your legitimate user should have. And the kube/config file has all your cluster connection details. If admins can quickly switch between clusters with no additional authentication for production, it's very easy to make a mistake, you think you're connected to the test cluster when you delete the database, but actually you've just wiped all your production data. Wired Brain Coffee's audit found exactly these problems. Using the default Kubernete's authentication with certificates valid for a year, kubeconfig files left on jump boxes for easy connection to clusters, no MFA anywhere. Again, classic symptoms of a team who are focused on delivery and not security. But Kubernetes has integration points for authenticating with external identity providers, and these issues are simple to fix.

### Focusing on Defense-in-depth

We've locked down cluster access for Wired Brain Coffee with MFA for end users, tied to the same RBAC modeling approach. Kubernete supports OpenID Connect integration, which means you can delegate authentication with pretty much any identity provider. Wired Brain run on AKS so they use Azure Entra ID and enforce security policies organization‑wide. Authentication and authorization are separate. You can model the security requirements for applications and users with roles and bindings, which live alongside your application spec. The team who owns the app decides on the access levels. Authentication for end users is handled outside of the cluster by your identity provider, that means you can enforce whatever policies your organization needs, MFA as a typical minimum. When kubectl sends the token onto the Kubernetes API server, it contains the authenticated user's identity and groups, that's used to find the roles and evaluate the permissions. Defense in Depth is a popular buzz phrase, but it's a really important mindset to get into. Does the application you're running even need access to the Kubernetes API? Every pod in your cluster is a potential attack vector. There might be a supply‑chain vulnerability hidden in a dependency, or you may be using an outdated image with known security issues. Compromises do happen and then RBAC becomes your containment system. The attacker can't jump namespaces, can't read secrets they shouldn't access, can't delete critical workloads, each permission boundary forces them to work harder, making more noise and increasing your chances of detecting them. Securing end‑user access is critical because users have legitimate reasons to be powerful, they debug production issues, deploy updates, investigate problems, but they may also get into bad habits, sharing credentials for convenience, fall for phishing attacks, MFA and temporary tokens mean a stolen laptop or leaked kube config is unlikely to give an attacker any access. And for production systems, you can use temporary elevation with an approval process, so users get admin permissions only when they need it. Many teams stop here thinking access control is enough, but can you trust the applications running inside the container? Protecting the Kubernetes API blocks a powerful attack vector, but if your apps get compromised, an attacker still has an entry point to your systems. The next layer in Defense in Depth is controlling what gets deployed in the first place, container scanning with security policies that prevent vulnerable or misconfigured workloads from running.

### Understanding Supply-chain Complexity

Container images are packaged with the complete software stack your application needs, but you need to manage that stack to keep it secured. Every container image is an entire software supply chain, which you don't control. Your Python application starts with a base image from Docker Hub, that builds on Debian or Alpine Linux, and that base image contains hundreds of packages maintained by different teams on independent release schedules. You add your framework, Django, Flask, Fast API, each with their own dependencies, then your application requirements, which pull in their requirements, creating a tree of code from dozens of sources. The Dockerfile hides a huge collection of release schedules. Debian and Python have predictable schedules with releases yearly or biannually. Frameworks and libraries release on their own schedule when they have new features or bug fixes. Each component follows its own timeline and can have emergency patches for a security fix, any changes could break your application. This challenge gets worse the longer you leave it. You scan your image when you deploy and find it's clean. For the next deployment, you scan again, and a critical CVE is discovered in a dependency. Your application is now vulnerable. You can update the library, but maybe that breaks compatibility with your framework. You can update the framework, but maybe that needs a newer version of Python. You update Python, and now half your other dependencies don't work. In my Plural sight course, Docker for Software Development: Deploying Apps, I show you how to use Trivy to scan images. Integrating that in your CI/CD pipeline is a great start, but a continuous challenge needs a continuous solution. Ongoing scanning is the key to managing this, continuous monitoring with a system component that checks all the images in your production cluster. The scanning database is regularly updated, you know when new CVEs are found, and you can patch critical issues as soon as they're discovered. You're making regular incremental updates, so it's much easier to keep all your dependencies in sync. At Wired Brain Coffee, the security audit flagged that they only scanned images on deployment. Checking all their production images again, they found critical vulnerabilities in everyone. Now they have a lot of updates to apply and regression test because the supply chain complexity overwhelmed their manual update processes. Continuous scanning of all container images would give them an audit log of known issues and an actionable set of fixes.

### Vulnerability Reports and Policies

Image scanning is a crucial part of your security story, and tools like Trivy Operator and Kyverno are Kubernetes native. The Trivy operator scans all container images for every pod and produces a vulnerability report. Because these are standard Kubernetes resources, you can automate your response to vulnerability reports. You could simply archive them to long‑term storage or logs as your audit trail, or you could trigger actions like creating tickets to address a new critical vulnerability. You can also use them for policy rules. Kyverno is a policy engine which lets you use cluster policies to restrict what can run in Kubernetes. ### Securing Communication with Network Policies

Defense in Depth is the key to securing Kubernetes, and the last piece is securing applications at runtime. We've built two layers of defense for Wired Brain Coffee, RBAC and MFA control, who can access the cluster and what they can do. Image scanning and admission policies, control what gets deployed, these reduce the likelihood of compromise. But there are still attack vectors in the running application, a supply chain attack or a zero‑day CVE gets discovered in an approved base image, or there's an insider threat, you can't assume there are no bad actors inside the organization. Runtime security limits the damage if there is a compromise. The main factors to secure are communication between components and the storage of sensitive configuration data, like database credentials or API keys. Kubernetes networking is open by default. Any pod in any namespace can communicate with any other pod. A compromised container is like a jump box which an attacker can use to go anywhere else in the cluster. Network policy is the solution, it's part of the standard Kubernetes API which acts as a firewall with traffic rules. Ingress specifies which pods can send traffic in, egress specifies which pods you can send traffic out to. Traffic that doesn't match and allow rule gets blocked. Network policies work on labels. A policy identifies the pods it applies to using a selector. Pods which are allowed ingress are identified by another selector, and pods which are allowed for egress have a selector. Typically, you start with a default deny policy in each namespace that uses an empty selector to match all pods. It's a failsafe because if someone forgets to apply a policy to a pod, it's effectively outside of the firewall. Then you add specific policies permitting only the traffic your application needs. For the Wired Brain app, the database should only allow ingress from the APIs, the API should only allow egress to the database, and ingress from the web app. The web app should only allow egress to the APIs.

### Why You Need the CSI Secrets Driver

Securing applications in Kubernetes is all about restricting what pods can do. Network policies now control traffic flow in the Wired Brain cluster. Each pod can only access its immediate downstream component. A compromised web pod can't reach the database directly or pods in any other namespace. It can't communicate outside of the cluster, so attackers can't exfiltrate data, that's solid security at the network layer. The other runtime security concern is protecting sensitive data, database credentials, API keys, and certificates. The Wired Brain team use the built‑in Kubanetes secret resource, but it's not a secure option. Kubanetes secrets store data as base64‑encoded strings, that's encoding, not encryption. Anyone with read access to secrets can decode them. Kubectl even has a helper method for that. The Wired Brain audit found database passwords that any developer could read, RBAC can prevent that by restricting secret access, but the data at rest is usually completely unprotected. The cluster database, etcd, can encrypt secrets at rest, but that requires additional configuration, which is often skipped. The proper solution is external secrets management. Sensitive data needs to live in a dedicated storage service outside of the cluster, that could be Azure Key Vault, AWS Secrets Manager, or HashiCorp Vault. Applications retrieve secrets at runtime, which get loaded into pods as volumes. The Secret store CSI driver powers this integration. You include a SecretProviderClass in your model, which is an abstraction over an external store. You specify the provider and the parameters to connect, then you explicitly state the secrets to fetch. In your Pod spec, you reference that ProviderClass as a volume, then you can mount the volume in your container and read the value from the secret store as a text file. Authentication to the secret store is seamless, without any cloud credentials in your model. You use annotations to link the Kubernetes service account to a cloud security principle, Azure Entra app registration, AWS IAM role, or Google Cloud service account, that's used to authenticate to the vault. The benefits go beyond encryption, centralized secrets across clusters with a single place to manage them, versioning of secrets and rotation without having to redeploy your apps, authorization rules separate to the RBAC in your cluster with audit logging for every access.

### Completing the Security Model

Wired Brain Coffee will pass their next security audit because they've implemented a full security model. The final part of that is runtime containment, network policy segment traffic, so pods can only communicate where they should, compromised pods can't pivot through the cluster. External secrets with the CSI Driver mean credentials aren't stored in etcd, they get loaded from a secure external vault. Every pod only sees the secrets it needs in the file system, so if there is a breach, the blast radius is contained. There's more that can be done at the container level in the pod spec, but that needs care because it can break your application. You can apply a seccompProfile to Linux containers to restrict dangerous system calls. And for individual containers, you can turn off Privilege Escalation, use a readOnlyFilesystem, run as a NonRoot user, and remove all unnecessary capabilities. But you'll need to apply those one at a time and test your app because they might cause a failure. Runtime security is complemented by deployment security. Image scanning will tell you if a CVE is found, but it's likely to be contained by the increased pod security, which can be enforced with policies in the cluster, and that deployment security is complemented by cluster security, ensuring only valid users have access and applying RBAC to limit what they can do. We've used a specific toolset at Wired Brain, Azure Entra, Kyverno, Trivy, and Key Vault, but the principles work across clouds and distributions. This is what Defense in Depth looks like, least privileged access controls for users and applications, ongoing vulnerability scanning, enforced as one of many security policies, restricted access to networks and secrets for containment if there is a compromise. Security shouldn't be seen as a checkbox, new vulnerabilities emerge and attack techniques evolve, and these are the foundations for proactive defense.
